# ADR 0001: Architecture-Deepening Public API Boundaries

The package had drifted toward parallel descriptions of the same workflows: string-based equilibrium requests versus typed problem objects, canonical parameter records versus legacy runtime payloads, generic versus reactive target-family summaries, and hand-waved capability claims versus executable evidence. We standardize on these boundaries: typed `EquilibriumProblem` objects are the authoritative non-reactive equilibrium problem surface and run through `mixture.solve_equilibrium(problem)`; `mixture.equilibrium(kind=...)` remains a convenience facade that adapts explicit non-reactive requests and owns reactive-specialized entrypoints; `ParameterSet` is the canonical parameter-family boundary and owns runtime payload emission through `to_runtime_dict()` plus dataset loading through `from_dataset(...)`; `TargetDataset.target_family_summaries()` defines the shared target-family summary shape reused by generic and reactive regression diagnostics; and package capability claims are derived from registered evidence surfaced by `epcsaft.capabilities()` and `capability_evidence`. This keeps the public API easier to navigate, keeps agent docs aligned with code, and prevents capability text from drifting ahead of production evidence.
